\begin{abstract}
Tool-using agents create idle gaps between turns, but long-context
inference systems usually compress KV caches before the next turn is
known.
That can leave a stale active cache: tokens that looked unimportant at
turn $N$ may become critical once a tool call returns or a follow-up
query is revealed.
We frame cross-turn KV cache repair as a runtime-systems and
benchmarking problem for adaptive inference.
To study it, we introduce \idlekv{}, a minimal idle-window repair
mechanism together with a matched-footprint protocol for testing whether
newly revealed turn-$(N{+}1)$ information can improve a compressed cache
at fixed final active decode-time footprint.
On a Qwen2.5-7B-Instruct prototype at 32K context, end-to-end repair
stays below 50\,ms through $K=500$ on the target hardware; this draft
reports that feasibility result and the protocol for the pending quality
study.
\end{abstract}

% -----------------------------------------------------------------------
\section{Introduction}
% -----------------------------------------------------------------------

\paragraph{Motivation.}
Adaptive inference is increasingly shaped by deployment constraints, and
tool-using agents make that pressure especially clear.
In batch-1 settings, decoding is interleaved with file reads, tests,
retrieval, remote APIs, and other external actions that pause
generation for milliseconds to seconds.
Those pauses are usually treated as dead time.
We argue that they expose a valuable new adaptation point: the idle
interval between turns.
We study a specific cross-turn case: revising a compressed cache after
new turn information arrives but before decoding resumes.

\paragraph{Adjacent work.}
On the systems side, PagedAttention/vLLM improves KV-memory efficiency
during serving~\citep{vllm}, while InferCept preserves paused requests
to avoid recomputing prefixes when augmented generation
resumes~\citep{infercept}.
On the efficient-decoding side, SnapKV, StreamingLLM, and H$_2$O
compress the cache under a fixed budget, while query-aware methods such
as QUEST emphasize that token criticality depends on the
query~\citep{snapkv,streamingllm,h2o,quest}.
These methods address adjacent parts of the problem, but not the
cross-turn case studied here.
Serving systems preserve interrupted state efficiently, while
query-aware compression still chooses the active cache before the next
turn is known.
By contrast, \idlekv{} starts from an already compressed paused state,
does not rerun selection over the full prefix, and revises memory only
within the idle-window latency budget before decoding resumes.

\paragraph{Problem setting.}
For long-context agents, relevance over a fixed context can shift across
turns.
A tool result or follow-up question can make earlier context newly
important, and recent analysis shows that KV compression can sharply
degrade instructions and facts that later become
important~\citep{pitfalls}.
The core failure is therefore temporal: the cache is allocated using the
wrong relevance signal for the next turn.
In practical terms, the system keeps what looked important last turn
instead of what will matter once the tool call returns.
Current benchmarks also leave this regime under-specified.
Single-turn long-context tasks can show compression loss, and serving
benchmarks can show pause/resume efficiency, but neither isolates
whether future-turn information can improve a compressed cache at fixed
final active decode-time footprint.
A useful benchmark for this setting needs explicit turn boundaries, a
hidden future query, shared pre-turn history, and matched-footprint
controls.

\paragraph{Approach and scope.}
\idlekv{} targets this gap with dynamic KV cache repair.
After turn $N$, the runtime compresses the evictable context portion to a
base budget, buffers evicted KV on CPU, waits for the tool call, and
during the idle window scores the buffer against the now-known
turn-$(N{+}1)$ query before restoring up to $K$ context tokens.
The adaptation happens in the active KV allocation itself, not in model
depth, routing, or weight choice.
This is state-revising pre-resume repair: unlike pause/resume serving,
it intentionally mutates memory before resumption; unlike within-turn
query-aware compression, it starts from a paused compressed state and
does not rerun full-prefix selection once $Q_2$ is revealed.
The practical question is whether tool-call time can be used to improve
the \emph{final} cache seen by the next turn at the same final active
decode-time footprint.
Concretely, the draft contributes:
\begin{itemize}[leftmargin=1.4em,noitemsep,topsep=2pt]
  \item a matched-footprint protocol for cross-turn relevance shift that
  isolates better cache selection from simply keeping more tokens; and
  \item a simple idle-window repair path with prototype latency
  measurements on one model, one eviction policy, and one hardware
  stack.
\end{itemize}
The paper does not yet claim matched-footprint turn-$(N{+}1)$ quality
gains.

\section{Protocol}
\label{sec:method}
% -----------------------------------------------------------------------

\paragraph{Overview.}
The key systems assumption is simple: a paused request need not preserve
its memory state exactly.
If the next turn is already known, the idle interval before resumption
can be used to improve the active cache that the next decode phase will
see.
We study the smallest such mechanism: repair buffered evictions without
rerunning a full query-aware compressor over the entire prefix.

\paragraph{Two-turn protocol.}
The protocol is simple.
The model first processes a long context and answers $Q_1$ from the full
cache.
After that answer, the runtime compresses only the evictable context
portion to a base budget $\bbase$, while preserving the exact $Q_1$
prompt and generated-answer tail.
During the subsequent tool-call idle window, the turn-$(N{+}1)$ query
$Q_2$ becomes known.
\idlekv{} encodes $Q_2$ against the active post-$Q_1$ cache, scores the
evicted context tokens against that signal, restores $K$ buffered
context tokens, and answers $Q_2$ from the repaired cache.
The protocol does not rerun full-prefix compression after $Q_2$ is
revealed.

\paragraph{Repair unit.}
Our prototype restores short query-scored local bursts rather than
isolated single tokens.
Here $K$ always denotes a restored-context-token budget, not an anchor
count.
Candidate anchor positions in the evicted buffer are ranked by the
future-query signal; accepted bursts are merged on overlap; and
restoration stops once $K$ context tokens have been reactivated.
For split-query retrieval tasks, this is the right unit of repair: the
query often contains a key-bearing anchor while the answer depends on
nearby value tokens.

\paragraph{Evaluation principle.}
All main claims are made against a no-repair baseline with the same
final active decode-time footprint.
Let $\bmatch$ denote the no-repair condition that applies the base
eviction policy with context budget $\bbase + K$ while preserving the
same $Q_1$ prompt and answer tail.
Because every compared run shares the same preserved tail and the same
active-token budget, the active cache footprint seen by $Q_2$ is
matched across $\bmatch$, \idlekv{}, and $\oraclek$, although the
selected context tokens differ by design.
Total retained CPU+GPU state is not matched, because \idlekv{} also
keeps an auxiliary CPU buffer of evicted KV.
We report:
\begin{equation}
  \mathrm{lift} =
  \mathrm{Score}(\mathrm{IdleKV}) - \mathrm{Score}(B_{\mathrm{match}})
\end{equation}
and
\begin{equation}
  \mathrm{recovery} =
  \frac{\mathrm{Score}(\mathrm{IdleKV})-\mathrm{Score}(B_{\mathrm{match}})}
       {\mathrm{Score}(\mathrm{Oracle}\mbox{-}K)-\mathrm{Score}(B_{\mathrm{match}})}.
\end{equation}
Here $\mathrm{Score}$ is the mean fraction of the two $Q_2$ needles
recovered correctly per example, averaged over the fixed evaluation set
shared across all conditions for that setting.
The numerator measures whether repair selects a better final cache than
matched no-repair retention; the denominator measures how much
recoverable headroom remains in the regime.
We report $\mathrm{recovery}$ only when
$\mathrm{Score}(\mathrm{Oracle}\mbox{-}K) >
\mathrm{Score}(B_{\mathrm{match}})$.
$\oraclek$ obeys the same $K$-token budget and the same burst-packing
rule as \idlekv{}, but ranks anchors using gold $Q_2$ needle locations
from the synthetic task metadata.
The protocol therefore tests whether newly revealed turn-$(N{+}1)$
information can improve a compressed cache relative to matched
no-repair retention.

\section{Setup}
\label{sec:setup}
% -----------------------------------------------------------------------

\paragraph{Primary setting.}
The main text focuses on one primary benchmark slice rather than a broad
matrix.
We use Qwen2.5-7B-Instruct~\citep{qwen} at 32K rendered context on a
single RTX PRO 6000 Blackwell GPU, with SnapKV~\citep{snapkv} as the
base eviction policy.
The task is a split-query version of \texttt{MQ-NIAH-4q} derived from
RULER~\citep{ruler}: one shared long context contains four needles,
$Q_1$ asks for needles 1 and 4, and $Q_2$ asks for needles 2 and 3.
This balanced split keeps two answers per turn while removing the
always-recency-kept tail needle from $Q_2$.
The resulting positive-control slice keeps future relevance hidden and
avoids giving the matched no-repair baseline half the answer for free.

\paragraph{Configuration.}
The planned primary study avoids recency-saturated settings, where
small budgets are dominated by recent tokens and query-aware scoring
cannot change the kept set.
We therefore use context-only SnapKV with a fixed context recency
reserve and evaluate:
\begin{itemize}[leftmargin=1.4em,noitemsep,topsep=2pt]
  \item context-only SnapKV with fixed recency reserve
  $R_{\mathrm{ctx}}=128$;
  \item base budget $\bbase=512$;
  \item concise split prompts that request the values only,
  comma-separated, with a short answer prefix and fixed
  $24$-token decode budget per split turn;
  \item pilot repair budgets $K \in \{100, 250, 500\}$, with the
  expectation that the main table will report the smallest informative
  subset; and
  \item one shared $Q_1$ transcript across all conditions and a fixed
  100-example evaluation set per setting.
\end{itemize}
The turn-$N$ eviction signal is the exact generated $Q_1$ answer tail,
not a generic last-$32$-token cache suffix.

\paragraph{Condition semantics.}
We first generate one full-cache turn-$N$ transcript comprising the
shared context, $Q_1$, and the model-generated $Q_1$ answer tail.
Every condition reuses that exact transcript when answering $Q_2$.
$A$ keeps the full context active during $Q_2$ and is therefore the
full-cache ceiling under the fixed transcript.
$B$ compresses only the evictable context portion to $\bbase$.
$\bmatch$ applies the same base eviction policy at context budget
$\bbase + K$ and performs no repair.
Random-$K$ and Oldest-$K$ start from $B$ and restore the same $K$-token
budget from the same evicted buffer without future-query scoring.
\idlekv{} repairs $B$ during the idle gap, and $\oraclek$ applies the
same burst-packing rule using gold $Q_2$ needle locations from the
synthetic task metadata.

\paragraph{Scope.}
The paper keeps the main text to one task family, one model, and one
eviction policy.
That is enough to answer the first question cleanly: once the future
query is revealed, can idle-time repair produce a better final cache
than matched no-repair retention?
This study estimates conditional turn-$(N{+}1)$ quality given a fixed,
full-cache-generated turn-$N$ transcript; it does not measure end-to-end
two-turn agent quality.
The main benchmark slice is a positive control chosen to keep future
relevance hidden, remove trivial recency wins from $Q_2$, and keep
eviction choices nontrivial, not to estimate average gains across agent
workloads.
It also does not yet compare against alternative post-$Q_2$ refresh
mechanisms that use the same idle window.

\section{Feasibility}
\label{sec:results}
% -----------------------------------------------------------------------

\paragraph{Current evidence.}
The current draft reports measured repair feasibility only.
The central quality question for the protocol remains the next
measurement: at fixed final active decode-time footprint, does
\idlekv{} improve turn-$(N{+}1)$ score over $\bmatch$?

\paragraph{Measured feasibility.}
Table~\ref{tab:latency} reports the current measured overheads on the
target hardware for the prototype repair path.

\begin{table}[t]
  \caption{Measured repair overhead on the target hardware.}
  \label{tab:latency}
  \centering
  \small
  \setlength{\tabcolsep}{4pt}
  \begin{tabular}{l c c c}
    \toprule
    $K$ & transfer p50 & total repair p50 & total repair p99 \\
    \midrule
    100 & 2.68\,ms  & 35.00\,ms & 37.34\,ms \\
    250 & 6.09\,ms  & 38.89\,ms & 40.44\,ms \\
    500 & 12.58\,ms & 46.94\,ms & 53.47\,ms \\
    \bottomrule
  \end{tabular}
\end{table}

Measured end-to-end repair stays below 50\,ms through $K=500$.
Separately, transfer p50 already rises smoothly to 25.86\,ms at
$K=1000$, which suggests room to extend the full end-to-end study to
larger repair budgets.
On the current prototype stack, the measured repair path stays well
below one tenth of a second.

\paragraph{Planned matched-footprint study.}
The next empirical step is the quality comparison itself.
The primary outcome is \emph{lift} over $\bmatch$ on turn-$(N{+}1)$
score.
\emph{Recovery} is secondary and is only reported when $\oraclek$
exceeds $\bmatch$.
Evidence for the mechanism would be a pattern in which \idlekv{}
exceeds $\bmatch$, Random-$K$, and Oldest-$K$, while $\oraclek$
remains above \idlekv{} whenever recoverable headroom exists.
A null or reversed result in a regime with positive oracle headroom
would point either to a weak repair signal or to a mismatched benchmark
slice.

\section{Discussion}
\label{sec:discussion}
% -----------------------------------------------------------------------

\paragraph{Position in adaptive inference.}
This paper is best read as a runtime-systems and benchmarking
contribution for adaptive inference.
Within that landscape, flexible architectures and routing change the
model path; test-time compute changes how much computation to spend;
compression and optimization reduce model or token cost; and efficient
decoding adapts attention or cache retention within a turn.
\idlekv{} points to a complementary question: once a request is paused
and compressed, can the runtime use the idle gap to revise memory before
the next turn begins?
That question matters most for agent workloads, because external actions
create exactly the pause in which such revision can happen.

\paragraph{Relation to serving systems.}
\idlekv{} is complementary to serving systems such as vLLM and
InferCept.
Those systems decide how to preserve, schedule, and resume interrupted
requests efficiently~\citep{vllm,infercept}; \idlekv{} asks what useful
quality-improving work can be done \emph{during} the interruption before
decoding resumes.
It is therefore state-revising pre-resume repair, not state-preserving
pause/resume serving.
It is also distinct from within-turn query-aware eviction: the request
has already been compressed, full-prefix recomputation is disallowed,
and the runtime operates under an idle-window latency budget.

\paragraph{When repair should help.}
Repair should help when new turn information changes relevance, the base
eviction policy still has real choice beyond recency, and the hindsight
ceiling leaves nontrivial headroom over $\bmatch$.
It should help less when the future turn adds little new information,
the keep budget is recency-saturated, or the idle gap is shorter than
the repair overhead.
These criteria matter both for benchmark design and for interpreting
null results.

\paragraph{Limitations and next steps.}
The study is intentionally conditional and favorable: one model, one
eviction policy, one synthetic split-query family, fixed
full-cache-generated turn-$N$ transcripts, and no alternative post-$Q_2$
refresh baselines yet.
That makes the mechanism legible, but it is not evidence about
end-to-end two-turn agent quality or average gains across workloads.
The immediate next steps are measured matched-footprint quality on the
primary slice, sham-query and low-headroom controls, a milder contiguous
split of the same task family as a robustness check, comparisons against
other post-$Q_2$ refresh mechanisms, and broader agent-style tasks in
which tool outputs, retrieved documents, or intermediate plans shift
relevance across multiple turns.
More broadly, the field still needs benchmarks with explicit idle
intervals, hidden future queries, matched no-repair controls, hindsight
ceilings, and latency measurements tied to real interruption windows.
If those studies continue to show gains, then idle-window memory
maintenance becomes a plausible systems primitive for long-context
agents.

% -----------------------------------------------------------------------
\section{Conclusion}
\label{sec:conclusion}
% -----------------------------------------------------------------------

Tool-call stalls in agent workflows can be treated as part of the
inference budget rather than as dead time.
\idlekv{} contributes a concrete way to spend that budget: revise a
compressed cache before the next turn under a matched final active
decode-time footprint.
The current draft establishes feasibility of the repair path on one
prototype stack and defines a protocol for measuring quality gains once
the full matched-footprint study is complete.
More broadly, long-context agent systems may need to be judged not only
by how efficiently they decode, but by how well they maintain memory
between turns.
