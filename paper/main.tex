\begin{abstract}
LLM agents in single-user deployments leave the GPU idle during every
tool call while relying on KV cache eviction to support long context
windows -- eviction that permanently discards tokens whose relevance
only becomes apparent once the tool result arrives.
We introduce \textbf{idle-window compute}: in batch-1 agentic inference,
tool-call idle cycles are an untapped adaptive compute budget for
improving next-turn quality.
We propose \textbf{IdleKV} as a first instantiation for KV cache repair:
evicted KV pairs are stored on CPU, scored against current query vectors
during the tool call, and the top $K$ (set adaptively by idle time, PCIe
bandwidth, and free memory) are restored before decoding resumes.
On Qwen2.5-7B-Instruct with SnapKV at 32K context, IdleKV achieves
mean repair recovery $\rho = 0.63$ against an oracle ceiling of 0.91,
closing 63\% of the eviction-induced solve-rate drop on SWE-bench
Verified at 12\,ms median overhead.
\end{abstract}

% -----------------------------------------------------------------------
\section{Introduction}
% -----------------------------------------------------------------------

\paragraph{Idle GPU time in batch-1 agents.}
A coding agent running on a single GPU operates with exactly one request
in flight.
When it calls \texttt{pytest}, the GPU executes no useful computation
until the result returns.
In multi-tenant serving, a scheduler fills that window with other users'
requests; in batch-1 deployments, no such queue exists.
Tool calls in agentic workloads span a wide range -- filesystem reads
complete in under 100\,ms while test suite runs routinely take
5--30\,s~\citep{swebench, continuum} -- and in all cases the GPU is idle
for the full duration.

\paragraph{KV eviction and delayed relevance.}
The same deployments use KV cache eviction to support long context
windows within a fixed GPU memory budget.
Methods such as SnapKV~\citep{snapkv}, StreamingLLM~\citep{streamingllm},
and H2O~\citep{h2o} drop tokens that score low on attention at eviction
time.
As~\citet{pitfalls} identify, this introduces a systematic failure mode:
instruction-level context -- system prompts, early-conversation facts,
multi-step reasoning chains -- can be evicted before its relevance fully
materialises in the attention signal, then permanently lost when a
subsequent tool result makes it the critical context.
Better eviction heuristics do not solve this; the tokens were genuinely
unimportant at the time they were dropped.

\paragraph{Idle-window compute.}
We observe that idle GPU time and eviction-induced quality loss are two
facets of the same deployment scenario, and propose using the former to
repair the latter.
We term this design principle \textbf{idle-window compute}: in batch-1
agentic inference, tool-call idle cycles constitute an adaptive per-turn
compute budget for improving next-turn context quality.
\textbf{IdleKV} is a first instantiation applied to KV cache repair.
When the model issues a tool call, evicted KV pairs are moved to a
CPU-resident buffer.
A background thread scores each entry against the current query vectors
and promotes the top $K$ back to the GPU before decoding resumes, with no
impact on user-visible latency.
The budget $K$ adapts to the available idle time, PCIe bandwidth, and
free GPU memory (Eq.~\ref{eq:budget}): a 30\,s pytest run supports
restoring thousands of tokens; a 100\,ms file read supports at most a few
dozen.

\paragraph{Contributions.}
Idle-window compute is not specific to KV repair — speculative
prefetching, cache defragmentation, and dynamic attention profiling are
all candidates — but we focus on KV repair as the most direct response
to the eviction failure mode in~\citet{pitfalls}.
Concretely, we contribute: \textbf{(1)}~\textbf{RULER-KVR}, a
three-condition benchmark with a new cross-turn retrieval task
(CT-MK-NIAH) isolating delayed-relevance failures;
\textbf{(2)}~an \textbf{oracle study} showing 91\% of degradation is
recoverable by content repair, bounding what any idle-window algorithm
can achieve; and \textbf{(3)}~\textbf{IdleKV}, which closes 63\% of
that gap at 4.2\,GFLOPs and 12\,ms median overhead, with ablations
showing selection quality matters more than repair volume.
SnapKV, StreamingLLM, and H2O compress but do not repair; IdleKV wraps
any of them as its base policy.
Speculative decoding~\citep{speculativedecoding} reclaims
intra-generation slack via draft proposals; our windows arise from
external I/O.
PagedAttention~\citep{vllm} addresses memory scheduling, not
content-level quality loss.

\section{IdleKV}
\label{sec:method}
% -----------------------------------------------------------------------

\paragraph{Setup.}
Let $\mathcal{C}^{(t)}$ be the active GPU KV cache at turn $t$ under
eviction policy $\pi$ with budget $B$.
At the start of each tool call, $\pi$ produces an eviction set
$\mathcal{E}^{(t)} \subset \mathcal{C}^{(t)}$; turn $t{+}1$ begins with
$\mathcal{C}^{(t)} \setminus \mathcal{E}^{(t)}$.
IdleKV selects a repair set $\mathcal{R} \subseteq \mathcal{E}^{(t)}$,
$|\mathcal{R}| \leq K$, and injects those KV pairs before decoding
resumes (Figure~\ref{fig:pipeline}).

\paragraph{Eviction buffer.}
At eviction time, IdleKV stores per-token records
$\{p,\, \mathbf{K}_p^{(l)},\, \mathbf{V}_p^{(l)},\,
\mathbf{q}_p^{(l)}\}_{l=1}^{L}$ in CPU pinned memory
(Qwen2.5-7B: 28 layers, 4 KV heads, head dim 128, fp16;
$\approx$0.5\,MB per token).
We compare four buffer-population strategies:
\textbf{L2-norm} -- retain tokens with highest $\|\mathbf{q}_p\|_2$,
our primary proposal (high query norm signals an active lookup target
at eviction time);
\textbf{attn-score} -- highest attention score at eviction;
\textbf{recency-inverse} -- oldest tokens first, targeting early-context
tokens preferentially dropped by SnapKV's observation window;
\textbf{random} -- ablation baseline.

\paragraph{Idle-time scoring.}
Given query window $\mathcal{W}$ (last $M$ active tokens), the relevance
of buffered token $p$ is:
\begin{equation}
  r_p = \max_{l}\; \max_{j \in \mathcal{W}}
    \frac{\mathbf{q}_j^{(l)} \cdot {\mathbf{k}_p^{(l)}}^\top}
         {\sqrt{d_{\mathrm{head}}}}
  \label{eq:score}
\end{equation}
The top-$K$ tokens by $r_p$ are transferred CPU$\to$GPU and injected at
their original sequence positions.

\paragraph{Adaptive budget.}
\begin{equation}
K = \min\!\Bigl(
  \bigl\lfloor (T_{\mathrm{idle}} - \epsilon) / t_{\mathrm{tok}} \bigr\rfloor,\;
  \bigl\lfloor M_{\mathrm{free}} / c_{\mathrm{tok}} \bigr\rfloor,\;
  |\mathcal{E}|
\Bigr)
\label{eq:budget}
\end{equation}
$\epsilon = 200$\,ms is reserved for injection startup;
$t_{\mathrm{tok}} \approx 1.8$\,ms/token (PCIe\,4.0 transfer plus
per-layer attention recompute, measured on target hardware);
$c_{\mathrm{tok}} \approx 0.5$\,MB.
At $T_{\mathrm{idle}} = 2$\,s: $K \approx 1{,}000$.
At the 14\,s pytest median: $K \approx 7{,}200$.

\paragraph{Implementation.}
We use Qwen2.5-7B-Instruct~\citep{qwen} via HuggingFace Transformers
with SnapKV (observation window 32, recency window 1024, sink size 4).
The repair thread runs in parallel with tool execution via non-blocking
pinned transfers; for $|\mathcal{E}| > 5{,}000$, a flat FAISS index
accelerates Eq.~\ref{eq:score} on CPU.

\section{Evaluation}
\label{sec:results}
% -----------------------------------------------------------------------

\paragraph{RULER-KVR.}
We introduce RULER-KVR, a modified version of RULER~\citep{ruler}
supporting three conditions on Qwen2.5-7B-Instruct with SnapKV
($k = 512$, 32K context, 250 examples per task):
\begin{itemize}[noitemsep,topsep=2pt,leftmargin=1.4em]
  \item[\textbf{A}] \textit{Monolithic}: single forward pass through full context and query.
  \item[\textbf{B}] \textit{Evict (no repair)}: KV cache dropped after prefill; model reprefills from scratch at query time.
  \item[\textbf{C}] \textit{IdleKV}: $K = 250$, synthetic $T_{\mathrm{idle}} = 5$\,s, $M = 64$.
\end{itemize}
Recovery rate: $\rho = (C - B)/(A - B)$.
Tasks: \textbf{S-NIAH} (single-needle retrieval); \textbf{CT-MK-NIAH}
(our new task -- target needle planted at turn~1, same-key distractor
injected post-eviction, query asks for original value; directly tests
the delayed-relevance failure mode under adversarial distraction);
\textbf{VT-2hop} (two-hop variable tracing); \textbf{FWE}
(frequent-word extraction).
An \textbf{oracle} condition restores all evicted tokens unconditionally,
establishing the $\rho$ ceiling.

\paragraph{Main results.}
Eviction reduces mean accuracy from 79.7\% to 45.9\%, a 33.8\,pp drop
(Table~\ref{tab:main}).
The oracle recovers to 76.8\% ($\rho_{\mathrm{oracle}} = 0.91$), showing
that 91\% of the gap is due to KV content loss rather than the two-call
structure itself -- we verify this with a round-trip test (serialise and
reload the intact cache, reproducing condition~A within 0.3\%).
IdleKV at $K = 250$ reaches 67.4\% ($\rho = 0.63$) at 4.2\,GFLOPs.

CT-MK-NIAH shows the largest absolute gain: the distractor error rate
falls from 61.1\% to 23.7\% ($\rho = 0.69$).
The target needle at depth~0 consistently scores highest under the
L2-norm probe after eviction, making it the most reliably identified
repair target.
VT-2hop ($\rho = 0.60$) and FWE ($\rho = 0.56$) show lower recovery
because both require distributed context that a fixed-$K$ budget
cannot fully restore.

\paragraph{Efficiency frontier.}
Figure~\ref{fig:frontier} plots mean $\rho$ against repair compute
across $K \in \{50, 100, 250, 500, 1000\}$.
L2-norm scoring Pareto-dominates all baselines at every compute level.
At 4.2\,GFLOPs, random selection reaches only $\rho = 0.31$ versus
L2-norm's 0.63: which tokens are restored matters more than how many.
Attn-score ($\rho = 0.55$ at the same budget) falls further behind at
larger $K$, as importance estimates from eviction time diverge from the
post-gap query distribution.

\paragraph{SWE-bench end-to-end.}
On 100 SWE-bench Verified tasks~\citep{swebench}, SnapKV eviction drops
the resolve rate from 18.4\% to 14.1\%.
IdleKV with dynamic $K$ per Eq.~\ref{eq:budget} recovers to 16.8\%,
closing 63\% of the 4.3\,pp drop.
Latency overhead: 12/34/71\,ms at p50/p90/p99; 98.3\% of repairs
complete before decoding resumes.
Consistent with Eq.~\ref{eq:budget}, gains concentrate on turns backed
by long tool calls (pytest, script execution) where the budget supports
restoring the tokens that matter.

\section{Discussion}
% -----------------------------------------------------------------------

\paragraph{Why L2-norm scoring works.}
Tokens with high query norm at eviction time are those the model was
actively using as lookup keys in the most recent context.
This signal predicts post-gap relevance better than attention scores from
eviction (which reflect the past query distribution) and better than
recency alone.
The residual 9\% oracle gap, unrecoverable by any content repair, is
attributable to RoPE position shift from the two-call structure.

\paragraph{Scope and limitations.}
Idle-window compute is specific to batch-1 inference; it offers no
benefit in batched serving where the GPU is never idle.
We evaluate one model (7B) and one eviction policy; generalisation to
larger models and other policies remains open.
Recovery is strongest for discrete-token retrieval and weakest for
distributed-state tasks, pointing to larger $K$ or multi-gap
accumulation as natural extensions.

\paragraph{The idle-window compute framework.}
The broader significance of this work is the design principle itself.
Batch-1 agentic inference is a distinct deployment regime -- one that is
increasingly common with local coding agents, on-device assistants, and
single-GPU research workloads -- and it exposes a compute resource that
conventional serving analysis overlooks.
KV repair is a first and practical use of that resource; we expect the
idle-window compute abstraction to motivate further work on adaptive
context management for agentic inference.
