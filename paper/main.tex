\begin{abstract}
Long-context agent workflows often pause between turns while
tools or users respond, but key-value (KV) cache compression is usually
a one-shot decision made before the next turn is known.
Each new turn can induce a relevance shift: a distributional change in
which historical tokens matter for the next decode; we frame pre-resume
cache repair as test-time adaptation of active KV state, where model
weights are fixed but the runtime revises which past context remains
active under a memory budget.
We evaluate repair against no-repair under the same number of active
context KV rows and introduce \idlekv{}, a pre-resume KV promotion
operator that retains evicted KV in a host-memory warm tier, scores it after
the turn-$(N{+}1)$ query $Q_2$ is known, and restores up to $K$ context
tokens before decoding resumes.
On Qwen2.5-7B-Instruct at 32K context, calibrated split-query
MQ-NIAH-4Q/6Q/8Q restore-budget sweeps
show \idlekv{} improving $Q_2$ score over matched no-repair by $0.755$,
$0.575$, and $0.435$ at $K=128$, while random/oldest-token restores
stay near the matched baseline.
A fixed five-turn revisit diagnostic further shows repeated repair
across relevance shifts, with non-initial-turn gain $0.542$ over matched
no-repair at $K=80$ and paired bootstrap interval $[0.458,0.620]$.
Together, the matched active-cache frontier, repeated-shift diagnostic, and
retention-rule checks support pre-resume KV repair as a useful
cache-state adaptation operator for a single-node prototype with active GPU
KV and a pre-materialized host-memory warm tier.
The current evidence is controlled rather than end-to-end agent validation,
and faster approximate scoring remains necessary for deployment.
\end{abstract}

\section{Introduction}
% -----------------------------------------------------------------------

\paragraph{Motivation.}
Long-context agents create a temporal mismatch between when the runtime
compresses key-value (KV) state and when the next-turn relevance signal
arrives.
The model's attention state must fit a limited active GPU cache even when
the application-visible context is much larger, so compression often
commits which historical tokens remain active before the next user turn,
tool result, test failure, or file change is known.
In batch-1, local-GPU, or dedicated-tenant agent workflows, decoding is
interleaved with these external actions, which pause generation for
milliseconds to seconds while the KV state remains useful.
Recent measurements make this slack concrete: web-environment
interaction can account for up to 53.7\% of web-agent latency, and
OS-level execution that includes tool calls and container/agent
initialization can account for 56--74\% of coding-agent task
latency~\citep{agent_efficiency,agentcgroup}.
Those pauses are usually treated as dead time, while cache compression is
treated as a one-shot active-cache decision.
We argue that the idle interval between turns exposes a test-time
adaptation point: state can be compressed to the active budget, retained
in a host-memory warm tier, and promoted back into the active cache when
new turn information changes the distribution of relevant context.
This problem is not limited to question answering: any next-turn text,
tool result, test failure, or user correction can change which past
context should remain active.
We study a minimal controlled instance of that problem: revising a
compressed cache after the next-turn query is revealed but before
decoding resumes.
This places \idlekv{} at the intersection of efficient inference and
test-time adaptation: as agent tasks become longer and more dynamic,
runtimes may need to adapt the model's active memory state across turns,
not only compress it.

\paragraph{Setting.}
For long-context agents, the relevant subset of a fixed context can
shift after the first answer as later turns introduce new evidence or
constraints.
Recent analysis shows that KV compression can sharply degrade
instructions and facts that later become important~\citep{pitfalls}.
The core failure is temporal: the cache is allocated before the
relevance signal for the next turn is known.
Current benchmarks can show compression loss or pause/resume efficiency,
but they do not isolate whether future-turn information can improve an
already-compressed cache under the same resumed active-cache budget.
A useful benchmark for this setting needs explicit turn boundaries, a
hidden future query, shared pre-turn history, and no-repair controls
matched on resumed-cache budget.
In our experiments, the future relevance signal is an explicit $Q_2$
retrieval query so that the effect is measurable; the intended workflow
claim is broader than query answering.

\paragraph{Contributions.}
\idlekv{} is a test-time adaptation operator for tiered KV state,
conditioned on the newly available next-turn signal.
After turn $N$, the runtime compresses the evictable context portion to a
base active budget and retains evicted rows in a host-memory warm store.
When the turn-$(N{+}1)$ signal arrives, \idlekv{} scores that store and
promotes up to $K$ context-token rows back into the active GPU cache
before decoding resumes.
This paper uses an explicit $Q_2$ retrieval query as the scoring signal
because it gives a controlled first test of whether post-compression
repair can exploit new relevance information.
Unlike pause/resume serving, \idlekv{} changes the resumed active cache;
unlike within-turn query-aware compression, it starts from a paused
compressed state and does not rerun full-prefix selection after $Q_2$ is
revealed.
We contribute:
\begin{itemize}[leftmargin=1.4em,noitemsep,topsep=2pt]
  \item a matched resumed active-cache-budget protocol for cross-turn
  relevance shift that isolates better cache selection from simply
  keeping more active tokens;
  \item calibrated evidence on pooled split-query multi-query
  needle-in-a-haystack (MQ-NIAH) 4Q/6Q/8Q evaluations, a five-turn
  revisit diagnostic, and three first-stage retention rules, where
  \idlekv{} substantially exceeds the matched no-repair baseline and
  content-agnostic restore controls;
  \item a repair prototype with transfer/reinjection feasibility results
  and an instrumented latency probe that separates cheap state movement
  and proxy scoring from the current exact Q$_2$ attention-query scorer.
\end{itemize}
This controlled scope identifies the repair mechanism on one primary model and
one synthetic task family, with targeted portability and retention-rule checks.
The adaptation-facing implication is that long-context agents may need
between-turn KV state updates, not only pre-generation pruning.

\section{Related Work}
\label{sec:related}
% -----------------------------------------------------------------------

SnapKV, StreamingLLM, and H$_2$O decide what to keep from the current
prefix under a fixed budget, while query-aware methods such as QUEST
emphasize that token criticality depends on the active
query~\citep{snapkv,streamingllm,h2o,quest}.
Long-context and KV-lifecycle benchmarks stress contexts from ordinary
long documents to 100K-token regimes, showing that
application-visible context length does not remove the need for
memory-management mechanisms~\citep{longbench,inftybench,ruler,scbench}.
SCBench makes the KV-cache lifecycle itself a benchmark target, and
ShadowKV shows that sparse KV reconstruction and offload can improve
long-context serving throughput~\citep{scbench,shadowkv}.
Fine-grained KV retrieval systems dynamically fetch query-relevant rows
from larger stores during active inference~\citep{fier,pariskv}.
These works motivate dynamic KV state, but they address adjacent stages:
serving systems preserve, route, or reload interrupted state, while
within-request query-aware and retrieval methods choose which KV pages or
tokens to load during active inference.
By contrast, \idlekv{} starts from an already compressed paused state,
does not rerun full-prefix selection after $Q_2$ is revealed, and
revises the active cache before decoding resumes under a matched
resumed-cache budget.

Serving systems provide the substrate for this question.
PagedAttention/vLLM improves GPU KV allocation and block management during
serving~\citep{vllm}, while InferCept preserves paused requests to avoid
recomputing prefixes when augmented generation resumes~\citep{infercept}.
Agent and tool-use benchmarks make external actions part of the workload,
including API calls, web interaction, repository edits, and interactive
tasks~\citep{toollm,webarena,swebench,agentbench}.
Agent-serving and KV-tiering systems reason about resume prefills,
offloading, cache reuse, and tiered storage across GPU, host memory,
storage, and network tiers~\citep{continuum,agentserve,lmcache,ttkv}.
These systems manage where KV state lives and how it is reused;
\idlekv{} asks which offloaded rows should re-enter the active cache when
a later turn reveals a new relevance signal.

\section{Method}
\label{sec:method}
% -----------------------------------------------------------------------

\paragraph{Overview.}
The method starts from the observation that a paused request need not
preserve its KV state exactly.
If the next turn is already known, the idle interval before resumption
can be used to improve the active cache that the next decode will see.
We study the smallest such mechanism: promote selected offloaded rows
without rerunning a full query-aware compressor over the entire prefix.

\paragraph{Two-turn protocol.}
The protocol proceeds in two turns.
The model first processes a long context and answers $Q_1$ from the full
cache.
After that answer, the runtime compresses only the evictable context
portion to a base budget $\bbase$, while preserving the exact $Q_1$
prompt and generated-answer tail.
During the subsequent tool-call idle window, the turn-$(N{+}1)$ query
$Q_2$ becomes known.
\idlekv{} encodes $Q_2$ against the active post-$Q_1$ cache, scores the
offloaded evicted context tokens against that signal, restores $K$
context tokens, and answers $Q_2$ from the repaired cache.
The data path therefore has two movements: evicted rows are offloaded
from the device-resident cache to a host-memory tier after $Q_1$, and
selected rows are copied back into the active device-resident cache
before $Q_2$ decoding.
The protocol does not rerun full-prefix compression after $Q_2$ is
revealed.
This isolates the adaptation operator for one paused request on one GPU:
a single host-memory tier, the full $Q_2$ signal known before resumed
decoding, and a pre-materialized evicted-KV store.
Scheduling, remote tiering, and retention contention are separate systems
questions.

\paragraph{Repair operation.}
The repair step scores evicted context positions against the newly
available turn-$(N{+}1)$ signal, packs the highest-scoring anchors into
short local bursts, and promotes the selected KV rows back into the
active cache before decoding resumes.
The prototype restores bursts rather than isolated single tokens, because
the query often contains a key-bearing anchor while the answer depends on
nearby value tokens.
Here $K$ always denotes a restored-context-token budget, not an anchor
count.
The exact scorer forms attention scores between RoPE-rotated $Q_2$ query
vectors and the union of active and evicted keys, applies the softmax over
that union, then averages the evicted-token mass over layers and heads
after max-pooling over $Q_2$ tokens.
Positions are ranked by this score, with the turn-$N$ score and absolute
position used only as deterministic tie-breakers.
Each anchor expands to a fixed local burst (two tokens left, twenty right);
duplicates are skipped and the ranked list backfills until $K$ unique
evicted context rows are selected.
The current prototype scores token rows, but the systems abstraction is
promotion over cached units; a systems implementation would batch these
promotions into exposed blocks or pages.

\paragraph{Matched-budget evaluation.}
All main claims are made against a no-repair baseline with the same
resumed active-cache budget.
A matched resumed-cache budget means that $Q_2$ decoding sees
the same number of active evictable-context tokens in addition to the
identical preserved $Q_1$ prompt and answer tail; it does not mean that
the runtime stores the same amount of offloaded KV state.
Let $\bmatch$ denote the no-repair condition that applies the base
eviction policy with context budget $\bbase + K$ while preserving the
same $Q_1$ prompt and answer tail.
\idlekv{} instead decodes from $\bbase+K$ active context rows by starting
from $\bbase$ retained rows and promoting $K$ rows from the host-memory
warm store; the extra store bytes, scorer time, and transfer time are reported
separately rather than hidden in the matched active-cache budget.
The main restore-budget frontier reports raw answer score so readers can
compare absolute quality across task difficulty levels.
For effect-size comparisons, we also report score gain over matched
no-repair,
\begin{equation}
  \Delta_{\mathrm{repair}} =
  \mathrm{Score}(\mathrm{IdleKV}) - \mathrm{Score}(B_{\mathrm{match}})
\end{equation}
where $\mathrm{Score}$ is the mean fraction of held-out $Q_2$ targets
recovered correctly per example, averaged over the fixed evaluation set
shared across all conditions for that setting.
For diagnostic purposes only, \spanref{} uses benchmark metadata:
it enumerates annotated $Q_2$ answer-span groups and selects the
highest-scoring subset whose token cost is at most $K$.
\spanref{} is exact over this small span-group family, but it is not an
optimum over all $K$-token cache states and is not used as an upper bound.
Thus the protocol tests whether newly revealed turn-$(N{+}1)$
information can improve a compressed cache relative to matched
no-repair retention, not whether the runtime simply keeps more active
tokens.

\section{Experimental Setup}
\label{sec:setup}
% -----------------------------------------------------------------------

\paragraph{Benchmark family.}
The main evaluation uses Qwen2.5-7B-Instruct~\citep{qwen} at 32K
rendered context on a single RTX PRO 6000 Blackwell GPU, with
context-only SnapKV-style retention as the initial post-$Q_1$
compressor applied before repair~\citep{snapkv}.
The task family is a split-query version of multi-query
needle-in-a-haystack (MQ-NIAH) derived from RULER~\citep{ruler}.
Each needle binds a key to a value inside a long distractor context, and
the model must return the values for the keys requested in the current
turn.
The 2Q, 4Q, 6Q, and 8Q variants use the same underlying task template;
the name denotes how many key/value needles are queried across the two
turns.
Larger variants ask for more values and use longer decode budgets, but
do not change the benchmark family.
The point of this synthetic family is control rather than realism: the
future-relevant spans are annotated, the second-turn relevance shift is
explicit, and every condition shares the same turn-$N$ history.
For each variant, we pool balanced, non-recency-favored partitions whose
$Q_2$ side excludes the tail-recent needles; Appendix
Figure~\ref{fig:app-partition-endpoints} summarizes the exact splits.

\paragraph{Evaluation configuration.}
The main study avoids both recency saturation and the all-zero baseline
regime found at smaller budgets.
Unless stated otherwise, reported evaluations use context-only SnapKV
with fixed context recency reserve $R_{\mathrm{ctx}}=128$.
We set $\bbase=8192$ for MQ-NIAH-2Q, $\bbase=16384$ for MQ-NIAH-4Q, and
$\bbase=18432$ for MQ-NIAH-6Q/8Q, sweep
$K \in \{8,16,24,32,48,64,80,96,128\}$, and reuse one shared
full-cache-generated $Q_1$ transcript across all conditions within each
setting.
This removes turn-$N$ error-cascade differences so that the frontier
isolates resumed-cache selection quality at turn $N{+}1$.
MQ-NIAH-2Q/4Q/6Q use fixed 100-example evaluation sets per split;
MQ-NIAH-8Q uses 24 examples per split over five partitions.
Prompt and decode settings are reported in the appendix.

\paragraph{Scorers.}
The turn-$N$ eviction signal is the exact generated $Q_1$ answer tail,
not a generic last-$32$-token cache suffix.
The main curves use an exact Q$_2$ attention-query scorer that extracts
the model's $Q_2$ self-attention query vectors after RoPE and scores
them against active and evicted keys.
The latency probe also measures a cheaper $Q_2$-appended-state proxy
scorer, reported separately from the exact quality curves.

\paragraph{Baselines and references.}
Every condition reuses the same full-cache-generated turn-$N$
transcript when answering $Q_2$.
We compare the full-cache reference $A$, the base compressed cache $B$,
matched no-repair $\bmatch$, Random-$K$ and Oldest-$K$ restore
controls, and \idlekv{}.
Random-$K$ and Oldest-$K$ restore from the same evicted-KV store as
\idlekv{} without future-query scoring, so they test whether gains come
from query-aware repair rather than from generic reinsertion.
The specificity study additionally uses StaleQ-$K$ (score with the
previous-turn query), WrongQ-$K$ (score with another example's $Q_2$),
and Refresh-buffered, a stronger $Q_2$-time reference that reselects the
whole $\bbase+K$ context budget from active plus offloaded rows without
full-prefix recomputation.

\section{Results}
\label{sec:results}
% -----------------------------------------------------------------------

\paragraph{Matched-budget frontier.}
Across the 2Q/4Q/6Q/8Q MQ-NIAH family,
\idlekv{} consistently beats matched no-repair at the same resumed active
KV budget.
Figure~\ref{fig:main-frontier} reports the raw $Q_2$ answer-score
frontier; Random-$K$ and Oldest-$K$ stay near matched no-repair in the
same evaluations.
MQ-NIAH-2Q saturates by $K=80$, while matched no-repair remains near
$0.06$.
The higher query-count settings separate the curves more gradually: at $K=96$,
\idlekv{} scores $0.910$ on 4Q, $0.989$ on 6Q, and $0.960$ on 8Q,
while matched no-repair scores $0.245$, $0.422$, and $0.546$,
respectively.
This shows that generic reinsertion from the same offloaded evicted-KV
store does not explain the effect.
Appendix diagnostics additionally report \spanref{}, which shows that
benchmark-labeled answer spans remain recoverable at intermediate
budgets; on MQ-NIAH-6Q at $K=48$, \spanref{} scores $0.994$ versus
$0.653$ for \idlekv{}.

\paragraph{Next-turn signal specificity.}
Repair gains are specific to the newly revealed current-turn signal, not
just to restoring additional cached tokens or using any query-like text.
At $K=48$, StaleQ-$K$ and WrongQ-$K$ remain near matched no-repair:
their gains are $0.028$ with intervals $[-0.021,0.083]$ and
$[-0.014,0.069]$, respectively.
\idlekv{} reaches gain $0.326$ with interval $[0.243,0.410]$.
The same panel also shows the method boundary.
Refresh-buffered, which reselects the whole resumed active budget from
the active cache plus offloaded evicted-KV store using $Q_2$ scores,
reaches raw score $1.000$ at this operating point.
Thus \idlekv{} is not the strongest possible $Q_2$-time store-backed
selector; it is an incremental repair primitive that leaves
room for stronger selection policies.

\paragraph{Repeated relevance shifts.}
The repair effect survives repeated controlled relevance shifts, not only
one follow-up query.
For this diagnostic, each post-priming turn appends the previous answer
to the running cache, recompresses the current context to $\bbase$, and
repairs using either the current turn query or a stale previous-turn
query.
On a fixed five-turn MQ-NIAH-8Q revisit schedule ($K=80$, $n=24$),
\idlekv{} gains $0.542$ over matched no-repair on non-initial turns with
paired bootstrap interval $[0.458,0.620]$; Random-$K$ and Oldest-$K$
gain only $0.010$ and $0.021$.
The strongest effect appears on the revisit turn; Turn 3 is retained as
a saturation check because matched no-repair is already high.
A current-query-only audit matches \idlekv{}, while stale-query-only
repair recovers part of the gap but remains clearly below current-query
repair.
Thus the figure supports repeated cue-conditioned adaptation under
controlled relevance shifts; end-to-end task validation requires
trace-based agent benchmarks.

\paragraph{Retention-rule sensitivity.}
The adaptation effect persists under two protocol-matched retention
heuristics beyond the primary SnapKV-style rule.
This matters because the first compression decision controls which rows
remain active and which evicted rows remain searchable at repair time.
To compare policies with different matched no-repair baselines, the
figure reports score gain over each policy's own matched baseline.
The SnapKV and accumulated-attention rules show large mid-to-high
$K$ gains, and the structurally different sink-plus-recent
rule also passes the gate: at $K=128$, \idlekv{} gains $0.431$ over
matched no-repair while Random-$K$/Oldest-$K$ remain at the matched
baseline.
These are protocol-matched first-stage retention heuristics inspired by
H$_2$O and StreamingLLM, not canonical reproductions of those systems.
The claim is therefore sensitivity to the first-stage retention rule, not
named-algorithm performance.

\paragraph{Robustness and mechanism.}
The pooled result hides real variation across query partitions, but the
high-$K$ result is not carried by one easy partition.
At $K=128$, all twelve balanced partitions pass the same direction check:
\idlekv{} beats $\bmatch$ and both content-agnostic restore controls.
The minimum score gain over $\bmatch$ is $0.585$ for MQ-NIAH-4Q and
$0.387$ for MQ-NIAH-6Q, and $0.292$ for MQ-NIAH-8Q.
Appendix Figure~\ref{fig:app-partition-endpoints} gives the
per-partition endpoints.
A same-protocol Llama-3.1-8B 6Q check gives a non-saturated
portability check rather than a model-family claim: with $n{=}24$ and
$B=16{,}384$, \idlekv{} improves from $\bmatch\approx0.36$ to
$0.80/0.90/0.98/1.00$ at $K=24/32/48/64$, while Random-$K$/Oldest-$K$
remain near $\bmatch$.

The restored tokens also move the final active cache toward the annotated
$Q_2$ answer spans.
Appendix Figure~\ref{fig:app-selection-diagnostic} shows that
\idlekv{} overlap rises with $K$ while $\bmatch$ and content-agnostic
controls stay near baseline.
Overlap is only a mechanism diagnostic: the values-only task can be
solved without recovering every annotated token.

\paragraph{Real-repository diagnostic.}
To check whether the repair effect is limited to synthetic MQ-NIAH, we
ran a controlled real-repository relevance-shift diagnostic over 48
callsite examples from 12 pinned repositories drawn from the SWE-bench
repository pool.
Repair sees only an answer-redacted event cue, and all conditions decode
the same final event-plus-question prompt.
At $K=192$, event-only \idlekv{} improves exact identifier accuracy from
$0.188$ for matched no-repair to $0.729$, while random, oldest,
stale-cue, wrong-event, and file-name-assisted controls remain between
$0.167$ and $0.188$.
A label-assisted locality reference reaches $0.896$, so we treat this as
preliminary evidence that this controlled real-repository setting
instantiates repairable KV-state failures, not as a solved real-code
benchmark.
A file-gated variant that restricts candidates to the event-named file
reaches $0.833$, reducing but not closing the gap to the label-assisted
reference.
The gain remains positive after excluding the cue-only hit or rows with
answer-token retention; Appendix
Figure~\ref{fig:app-real-repo-diagnostic} gives the full diagnostic and
sensitivity checks.

\paragraph{Latency accounting.}
We separate quality-linked scorer diagnostics from systems-capacity
measurements.
Appendix Figure~\ref{fig:app-transfer-runtime} plots the fixed-$K$
latency/quality probe: the exact scorer used in the quality curves is a
research diagnostic, with $K=96$ p50 $6.08$--$7.62$\,s across
MQ-NIAH-2Q/4Q/6Q/8Q; the $Q_2$-appended-state proxy reduces paired
4Q/6Q p50 to $0.83$/$0.71$\,s.
In the controlled proxy frontier
(Appendix Figure~\ref{fig:app-proxy-controlled}), proxy \idlekv{} still
separates from matched no-repair and content-agnostic restores: at
$K=96$, it scores $0.970$/$0.894$ on 4Q/6Q versus $\bmatch$
$0.245$/$0.422$, while Random-$K$/Oldest-$K$ improve by at most
$0.010$/$0.004$ over $\bmatch$.
The readiness audit retains $1.09{\times}$/$0.83{\times}$ of the exact
$K=96$ repair gain for 4Q/6Q, so the proxy is evidence for a cheaper
scoring path in this benchmark rather than a production deployment claim.
The systems question is whether a scalable score/select/promote path can
fit inside seconds-scale pause budgets, not whether the current exact scorer
is deployable.

\paragraph{Capacity envelope.}
Figure~\ref{fig:runtime-envelope} reports a single-node p95 envelope:
a chunked GPU selector streams Qwen-shaped BF16 key rows from pinned
host memory, scores a length-64 query, selects $K$ rows, and adds the
measured 32K-active move-and-inject p95 for the same $K$.
Let $N$ denote offloaded candidate rows, $q$ query tokens, and
$B_{\mathrm{act}}$ active rows before promotion; the profiled service
time decomposes as
\[
T_{\mathrm{repair}}\!\approx\!
T_{\mathrm{scan}}(N,q)+T_{\mathrm{topK}}(N,K)+
T_{\mathrm{promote}}(K,B_{\mathrm{act}}).
\]
For fixed model shape and $q$, candidate scanning is linear in $N$;
top-$K$ and promotion are measured separately rather than assumed
constant.
Across $K\in\{96,512,1024,5000\}$, the length-64 p95 curve changes by at
most $6.1$\,ms; at $K=5000$, p95 grows from $50.0$\,ms at 32K
candidates to $1.20$\,s at 1M and $4.64$\,s at 4M, where 4M rows
correspond to about 240\,GB of full offloaded Qwen-shaped BF16 KV.
Thus candidate-store size dominates this path, while top-$K$ selection
and selected-KV copy/reinsertion are smaller measured components in this
probe.
The values are consistent with multi-second tool/environment components
reported in agent studies~\citep{agent_efficiency,agentcgroup}, but the
plot is a capacity envelope, not a trace-scheduled latency distribution.

\section{Discussion and Limitations}
\label{sec:discussion}
% -----------------------------------------------------------------------

\idlekv{} is not a standalone serving stack; it is a test-time
adaptation operator over active KV state for resource-adaptive tiered-KV
runtimes.
It complements systems that preserve, route, offload, or resume paused
requests efficiently~\citep{vllm,infercept,lmcache,ttkv,continuum,
agentserve}: here the question is how the runtime should adapt active
memory state during the interruption after the request has already been
compressed.
Repair should help when new turn information changes relevance, the base
eviction policy still has real choice beyond recency, and the idle gap
covers repair overhead.
The evidence remains controlled: one primary model, one primary retention
rule, a synthetic split-query family, and fixed full-cache-generated
turn-$N$ transcripts; the real-repository diagnostic is supporting
external-validity evidence, not a SWE-bench issue-resolution benchmark.
The next systems experiments are trace-scheduled repair against real wait
distributions, stronger learned or page-level selectors,
latency-normalized $Q_2$-time reselection, and full-prefix recompute
baselines.

\paragraph{Conclusion.}
Pauses between agent turns can be treated as part of the inference budget.
Across MQ-NIAH-4Q/6Q/8Q, \idlekv{} outperforms matched no-repair
retention while random and oldest-token restores stay near baseline,
showing that long-context agent systems should be judged by active-state
adaptation as well as decode efficiency.

\section{Additional Evaluation Details}
\label{app:evaluation-details}

\paragraph{Protocol details.}
Concise split prompts request values only, comma-separated, with a short
answer prefix and fixed decode budgets of 24 tokens for MQ-NIAH-4Q,
48 tokens for MQ-NIAH-6Q, and 64 tokens for MQ-NIAH-8Q.
Generation is deterministic greedy decoding with no sampling.
The main examples use dataset seed offset 0; Random-$K$ uses a
deterministic example- and $K$-dependent seed.
Bootstrap intervals are percentile intervals over paired
example-partition rows with fixed seeds.
Main Qwen runs use BF16 weights on one RTX PRO 6000 Blackwell GPU with
Python 3.12.3, PyTorch 2.10.0+cu128, and Transformers 5.2.0.
The exact Q$_2$ attention-query scorer extracts the model's actual
self-attention query vectors for $Q_2$ prompt tokens after RoPE and
scores them against active and evicted keys; this is distinct from
decode-time answer-token attention.
The $Q_2$-appended-state proxy instead appends $Q_2$ to the compressed
cache with an ordinary forward pass and reuses the appended $Q_2$ cache
rows as scoring rows.

\paragraph{Span-group diagnostic.}
\spanref{} starts from the same $B$ state and offloaded evicted-KV store
as \idlekv{}.
It enumerates the small family of annotated $Q_2$ answer-span-group
subsets, one key/value span group per requested needle, evaluates each
subset once under actual $Q_2$ generation, and chooses the
highest-scoring subset with total cost at most $K$.
It is exact over this benchmark-specific span-group family, but not over
all arbitrary token subsets or possible final generations.
Because the selected span-group subset may use fewer than $K$ rows, a
repair selector can occasionally score higher at low budgets by spending
the full budget on useful local neighborhoods around future-relevant
spans. Thus \spanref{} is a diagnostic span-group reference, not a
ceiling.

\paragraph{Partition choices.}
For MQ-NIAH-4Q, the pooled panel uses
$14{\rightarrow}23$, $24{\rightarrow}13$, and
$34{\rightarrow}12$.
These are the balanced $2|2$ partitions whose $Q_2$ side excludes the
tail-anchored fourth needle.
For MQ-NIAH-6Q, the pooled panel uses
$156{\rightarrow}234$, $256{\rightarrow}134$,
$356{\rightarrow}124$, and $456{\rightarrow}123$, so the second turn
excludes the two latest needles.
For MQ-NIAH-8Q, the pooled panel uses
$5678{\rightarrow}1234$, $1678{\rightarrow}2345$,
$2678{\rightarrow}1345$, $3678{\rightarrow}1245$, and
$4678{\rightarrow}1235$, again excluding the two latest needles from the
second turn.
Within-turn order is not treated as a separate condition because the
prompt asks for comma-separated values only and scoring checks whether
each requested value is recovered.
Recency-favorable splits such as $12{\rightarrow}34$ are left to
separate diagnostics rather than pooled into the main future-query
repair result.

\section{Additional Discussion}
\label{app:additional-discussion}

\paragraph{Tiered-cache scaling.}
Repair creates a two-level retention problem because the main curves match
only the resumed active GPU KV budget, not total host-plus-device storage.
A strict device-resident policy chooses the active KV budget, while a
looser off-device policy decides which evicted rows remain searchable,
compressed, summarized, or recomputable; in the Qwen-shaped accounting
above, 4M searchable rows already imply about 240\,GB before metadata.
Thus \idlekv{} supports the middle operation in a tiered hierarchy:
turn-conditioned promotion from a warm tier during an idle gap.
This is a hardware-facing implication, not a hardware result.

\paragraph{Future benchmark axes.}
Future benchmarks should vary the number, timing, and conflict structure
of new-turn constraints.
The 2Q and 8Q breadth checks suggest that easy regimes can saturate while
stricter multi-query turns expose how quickly eviction failures
accumulate.
The five-turn diagnostic strengthens the dynamic-workflow evidence, but
it remains synthetic; broader task families should keep the same
matched-budget controls while adding real tool traces and stronger
selector families.
