\begin{abstract}
Long-context agent workflows often pause between turns while
tools or users respond, but key-value (KV) cache compression is usually
a one-shot decision made before the next turn is known.
When the follow-up turn changes which earlier context is relevant, the
resumed active cache can be stale even at the same active-cache budget as
a no-repair baseline.
We formulate this as a matched resumed active-cache-budget problem and
introduce \idlekv{}, a pre-resume KV promotion operator that retains
evicted KV in an off-device tier, scores it after the turn-$(N{+}1)$
query $Q_2$ is known, and restores up to $K$ context tokens before
decoding resumes.
On Qwen2.5-7B-Instruct at 32K context, calibrated split-query
MQ-NIAH-4Q/6Q/8Q restore-budget sweeps
show \idlekv{} improving $Q_2$ score over matched no-repair by $0.755$,
$0.575$, and $0.435$ at $K=128$, while random/oldest-token restores
stay near the matched baseline.
A locked five-turn revisit diagnostic further shows repeated repair
across relevance shifts, with non-initial-turn gain $0.542$ over matched
no-repair at $K=80$ and paired bootstrap interval $[0.458,0.620]$.
These controlled diagnostics do not establish end-to-end agent gains,
but they support pre-resume cache promotion as a useful operator for
tiered-KV runtimes; faster approximate scoring remains necessary for
deployment.
\end{abstract}

\section{Introduction}
% -----------------------------------------------------------------------

\paragraph{Motivation.}
Long-context LLM inference often becomes a key-value (KV) cache
management problem: the model's attention state must fit a limited active
GPU cache even when the application-visible context is much larger.
In batch-1, local-GPU, or dedicated-tenant agent workflows, decoding is
interleaved with file reads, tests, retrieval, remote APIs, and other
external actions that pause generation for milliseconds to seconds while
the KV state remains useful.
Recent measurements make this slack concrete: web-environment
interaction can account for up to 53.7\% of web-agent latency, and
OS-level execution that includes tool calls and container/agent
initialization can account for 56--74\% of coding-agent task
latency~\citep{agent_efficiency,agentcgroup}.
Those pauses are usually treated as dead time, and cache compression is
usually treated as a one-shot active-cache decision.
We argue that the idle interval between turns exposes a different cache
management point: state can be compressed to the active budget, retained
in an off-device tier, and promoted back into the active cache when new
turn information arrives.
The broader systems problem is not limited to question answering: any
next-turn text, tool result, test failure, or user correction can change
which past context should remain active.
We study a minimal controlled instance of that problem: revising a
compressed cache after the next-turn query is revealed but before
decoding resumes.

\paragraph{Setting.}
For long-context agents, the relevant subset of a fixed context can
shift after the first answer as later turns introduce new evidence or
constraints.
Recent analysis shows that KV compression can sharply degrade
instructions and facts that later become important~\citep{pitfalls}.
The core failure is temporal: the cache is allocated before the
relevance signal for the next turn is known.
Current benchmarks can show compression loss or pause/resume efficiency,
but they do not isolate whether future-turn information can improve an
already-compressed cache under the same resumed active-cache budget.
A useful benchmark for this setting needs explicit turn boundaries, a
hidden future query, shared pre-turn history, and no-repair controls
matched on resumed-cache budget.
In our experiments, the future relevance signal is an explicit $Q_2$
retrieval query so that the effect is measurable; the intended workflow
claim is broader than query answering.

\paragraph{Contributions.}
\idlekv{} is a KV promotion operator for tiered-cache runtimes,
conditioned on the newly available next-turn signal.
After turn $N$, the runtime compresses the evictable context portion to a
base active budget and retains evicted KV in an off-device evicted-KV
store, implemented here as host memory.
When the turn-$(N{+}1)$ signal arrives, \idlekv{} scores that store and
promotes up to $K$ context-token rows back into the active GPU cache
before decoding resumes.
This paper uses an explicit $Q_2$ retrieval query as the scoring signal
because it gives a controlled first test of whether post-compression
repair can exploit new relevance information.
Unlike pause/resume serving, \idlekv{} changes the resumed active cache;
unlike within-turn query-aware compression, it starts from a paused
compressed state and does not rerun full-prefix selection after $Q_2$ is
revealed.
We contribute:
\begin{itemize}[leftmargin=1.4em,noitemsep,topsep=2pt]
  \item a matched resumed-cache-budget protocol for cross-turn
  relevance shift that isolates better cache selection from simply
  keeping more active tokens;
  \item calibrated evidence on pooled split-query multi-query
  needle-in-a-haystack (MQ-NIAH) 4Q/6Q/8Q evaluations, where \idlekv{}
  substantially exceeds the matched
  no-repair baseline and content-agnostic restore controls;
  \item a repair prototype with transfer/reinjection feasibility results
  and an instrumented latency probe that separates cheap state movement
  and proxy scoring from the current exact Q$_2$ attention-query scorer.
\end{itemize}
The empirical claim remains narrow: one model, one primary first-stage
retention rule in the main evaluation, one synthetic task family, and calibrated
settings within that family.
Its broader implication is systems-facing: long-context agents may need
between-turn KV cache maintenance, not only pre-generation pruning.

\section{Related Work}
\label{sec:related}
% -----------------------------------------------------------------------

On the systems side, PagedAttention/vLLM improves GPU KV allocation and
block management during serving~\citep{vllm}, while InferCept preserves
paused requests to avoid recomputing prefixes when augmented generation
resumes~\citep{infercept}.
Agent and tool-use benchmarks make external actions part of the
workload, including API calls, web interaction, repository edits, and
interactive tasks~\citep{toollm,webarena,swebench,agentbench}.
These benchmarks motivate paused, stateful workflows, while
agent-serving systems reason about resume prefills, offloading, and KV
retention across turn gaps~\citep{continuum,agentserve}.
Recent KV systems treat cache state as block-oriented infrastructure:
LMCache and TTKV expose tiered KV storage or temporal tiering across GPU,
host memory, storage, and network tiers, while serving runtimes combine
offload, cache reuse, and cache-aware scheduling~\citep{lmcache,ttkv,
continuum,agentserve}.
These substrates manage where KV state lives and how it is reused.
\idlekv{} is complementary: it studies a per-request promotion policy for
deciding which offloaded rows should re-enter the active cache when a
later turn reveals a new relevance signal.

On the efficient-decoding side, SnapKV, StreamingLLM, and H$_2$O decide
what to keep from the current prefix under a fixed budget, while
query-aware methods such as QUEST emphasize that token criticality
depends on the active query~\citep{snapkv,streamingllm,h2o,quest}.
Long-context and KV-lifecycle benchmarks stress contexts from ordinary
long documents to 100K-token regimes, showing that
application-visible context length does not remove the need for
memory-management mechanisms~\citep{longbench,inftybench,ruler,scbench}.
SCBench makes the KV-cache lifecycle itself a benchmark target, and
ShadowKV shows that sparse KV reconstruction and offload can improve
long-context serving throughput~\citep{scbench,shadowkv}.
These works motivate dynamic KV state, but they address adjacent stages:
serving systems preserve, route, or reload interrupted state, while
within-request query-aware methods choose which KV pages or tokens to
load during active inference.
By contrast, \idlekv{} starts from an already compressed paused state,
does not rerun full-prefix selection after $Q_2$ is revealed, and
revises the active cache before decoding resumes under a matched
resumed-cache budget.

\section{Method}
\label{sec:method}
% -----------------------------------------------------------------------

\paragraph{Overview.}
The method starts from the observation that a paused request need not
preserve its KV state exactly.
If the next turn is already known, the idle interval before resumption
can be used to improve the active cache that the next decode will see.
We study the smallest such mechanism: promote selected offloaded rows
without rerunning a full query-aware compressor over the entire prefix.

\paragraph{Two-turn protocol.}
The protocol proceeds in two turns.
The model first processes a long context and answers $Q_1$ from the full
cache.
After that answer, the runtime compresses only the evictable context
portion to a base budget $\bbase$, while preserving the exact $Q_1$
prompt and generated-answer tail.
During the subsequent tool-call idle window, the turn-$(N{+}1)$ query
$Q_2$ becomes known.
\idlekv{} encodes $Q_2$ against the active post-$Q_1$ cache, scores the
offloaded evicted context tokens against that signal, restores $K$
context tokens, and answers $Q_2$ from the repaired cache.
The data path therefore has two movements: evicted rows are offloaded
from the device-resident cache to a host-memory tier after $Q_1$, and
selected rows are copied back into the active device-resident cache
before $Q_2$ decoding.
The protocol does not rerun full-prefix compression after $Q_2$ is
revealed.
It is intentionally a mechanism study: one paused request on one GPU, a
single host-memory tier, the full $Q_2$ signal known before resumed
decoding, and a pre-materialized evicted-KV store.
Distributed-serving concerns such as routing, request migration,
queueing, remote or disk fetch, overlap with resumed generation, and
off-device retention contention are left to future systems work.

\paragraph{Repair operation.}
The repair step scores evicted context positions against the newly
available turn-$(N{+}1)$ signal, packs the highest-scoring anchors into
short local bursts, and promotes the selected KV rows back into the
active cache before decoding resumes.
The prototype restores bursts rather than isolated single tokens, because
the query often contains a key-bearing anchor while the answer depends on
nearby value tokens.
Here $K$ always denotes a restored-context-token budget, not an anchor
count.
The current prototype scores token rows, but the systems abstraction is
promotion over cached units; a systems implementation would batch these
promotions into its exposed blocks or pages.

\paragraph{Matched-budget evaluation.}
All main claims are made against a no-repair baseline with the same
resumed active-cache budget.
A matched resumed-cache budget means that $Q_2$ decoding sees
the same number of active evictable-context tokens in addition to the
identical preserved $Q_1$ prompt and answer tail; it does not mean that
the runtime stores the same amount of offloaded KV state.
Let $\bmatch$ denote the no-repair condition that applies the base
eviction policy with context budget $\bbase + K$ while preserving the
same $Q_1$ prompt and answer tail.
The main restore-budget frontier reports raw answer score so readers can
compare absolute quality across task difficulty levels.
For effect-size comparisons, we also report score gain over matched
no-repair,
\begin{equation}
  \Delta_{\mathrm{repair}} =
  \mathrm{Score}(\mathrm{IdleKV}) - \mathrm{Score}(B_{\mathrm{match}})
\end{equation}
where $\mathrm{Score}$ is the mean fraction of held-out $Q_2$ targets
recovered correctly per example, averaged over the fixed evaluation set
shared across all conditions for that setting.
$\goldk$ is a benchmark-metadata hindsight reference over annotated
$Q_2$ span groups; it is not a mathematical upper bound over arbitrary
token subsets.
Thus the protocol tests whether newly revealed turn-$(N{+}1)$
information can improve a compressed cache relative to matched
no-repair retention, not whether the runtime simply keeps more active
tokens.

\section{Experimental Setup}
\label{sec:setup}
% -----------------------------------------------------------------------

\paragraph{Benchmark family.}
The main evaluation uses Qwen2.5-7B-Instruct~\citep{qwen} at 32K
rendered context on a single RTX PRO 6000 Blackwell GPU, with
a context-only SnapKV-style first-stage retention rule~\citep{snapkv}.
The task family is a split-query version of multi-query
needle-in-a-haystack (MQ-NIAH) derived from RULER~\citep{ruler}.
Each needle binds a key to a value inside a long distractor context, and
the model must return the values for the keys requested in the current
turn.
The 4Q, 6Q, and 8Q variants use the same underlying task template; the
larger variants ask for more queried values and use longer decode
budgets, but do not change the benchmark family.
The names 4Q/6Q/8Q mean that four, six, or eight key/value needles are
queried across the two turns.
For each variant, we pool balanced partitions whose $Q_2$ side excludes
the tail-recent needles: three partitions for MQ-NIAH-4Q, four for
MQ-NIAH-6Q, and five for MQ-NIAH-8Q.
This avoids a trivial recency advantage for the no-repair baseline while
keeping the future-query mechanism measurable; Appendix
Figure~\ref{fig:app-partition-endpoints} summarizes the exact splits.

\paragraph{Evaluation configuration.}
The main study avoids both recency saturation and the all-zero baseline
regime found at smaller budgets.
Unless stated otherwise, reported evaluations use context-only SnapKV
with fixed context recency reserve $R_{\mathrm{ctx}}=128$.
We set $\bbase=16384$ for MQ-NIAH-4Q and $\bbase=18432$ for
MQ-NIAH-6Q/8Q, sweep
$K \in \{8,16,24,32,48,64,80,96,128\}$, and reuse one shared
full-cache-generated $Q_1$ transcript across all conditions within each
setting.
This removes turn-$N$ error-cascade differences so that the frontier
isolates resumed-cache selection quality at turn $N{+}1$.
MQ-NIAH-4Q/6Q use fixed 100-example evaluation sets per split; MQ-NIAH-8Q
uses 24 examples per split over five partitions.
Prompt and decode settings are reported in the appendix.

\paragraph{Scorers.}
The turn-$N$ eviction signal is the exact generated $Q_1$ answer tail,
not a generic last-$32$-token cache suffix.
The main curves use an exact Q$_2$ attention-query scorer that extracts
the model's $Q_2$ self-attention query vectors after RoPE and scores
them against active and evicted keys.
The latency probe also measures a cheaper $Q_2$-appended-state proxy
scorer, reported separately from the exact quality curves.

\paragraph{Baselines and references.}
Every condition reuses the same full-cache-generated turn-$N$
transcript when answering $Q_2$.
We compare the full-cache reference $A$, the base compressed cache $B$,
matched no-repair $\bmatch$, Random-$K$ and Oldest-$K$ restore
controls, \idlekv{}, and $\goldk$.
Random-$K$ and Oldest-$K$ restore from the same evicted-KV store as
\idlekv{} without future-query scoring, so they test whether gains come
from query-aware repair rather than from generic reinsertion.

\section{Results}
\label{sec:results}
% -----------------------------------------------------------------------

\paragraph{Matched-budget frontier.}
Across the 2Q/4Q/6Q/8Q MQ-NIAH family,
\idlekv{} consistently beats matched no-repair at the same resumed active
KV budget.
Figure~\ref{fig:main-frontier} reports the raw $Q_2$ answer-score
frontier; Random-$K$ and Oldest-$K$ stay near matched no-repair in the
same evaluations.
MQ-NIAH-2Q saturates by $K=80$, while matched no-repair remains near
$0.06$.
The higher query-count settings separate the curves more gradually: at $K=96$,
\idlekv{} scores $0.910$ on 4Q, $0.989$ on 6Q, and $0.960$ on 8Q,
while matched no-repair scores $0.245$, $0.422$, and $0.546$,
respectively.
This shows that generic reinsertion from the same offloaded evicted-KV
store does not explain the effect.
The $\goldk$ reference shows remaining benchmark-metadata selection
headroom at intermediate budgets, especially on MQ-NIAH-6Q at $K=48$
($0.994$ versus $0.653$ for \idlekv{} in absolute score).

\paragraph{Next-turn signal specificity.}
Repair gains are specific to the newly revealed current-turn signal, not
just to restoring additional cached tokens or using any query-like text.
At $K=48$, stale-query and donor-query repairs remain near matched
no-repair (gain $0.028$), while \idlekv{} reaches gain $0.326$
with a positive paired confidence interval.
The same panel also shows the method boundary.
Refresh-buffered, which reselects the whole resumed active budget from
the active cache plus offloaded evicted-KV store using $Q_2$ scores,
reaches the
$\goldk{}$ reference at this operating point.
Thus \idlekv{} is not the strongest possible $Q_2$-time store-backed
selector; it is an incremental repair primitive that leaves
room for stronger selection policies.

\paragraph{Repeated relevance shifts.}
The repair effect survives repeated controlled relevance shifts, not only
one follow-up query.
On a locked five-turn MQ-NIAH-8Q revisit schedule ($K=80$, $n=24$),
\idlekv{} gains $0.542$ over matched no-repair on non-initial turns with
paired bootstrap interval $[0.458,0.620]$; Random-$K$ and Oldest-$K$
gain only $0.010$ and $0.021$.
The strongest effect appears on the revisit turn, where \idlekv{} gains
$0.938$ with interval $[0.875,1.000]$.
Turn 3 has no matched-to-\goldk{} headroom and therefore dilutes, rather
than inflates, the aggregate.
A current-query-only repair matches \idlekv{}, while a stale-query-only
repair recovers part of the gap; current-query-only still exceeds
stale-query-only by $0.307$ with paired interval $[0.240,0.370]$.
Thus the figure supports repeated cue-conditioned adaptation under
controlled relevance shifts, but not end-to-end agent validation.

\paragraph{First-stage retention sensitivity.}
The repair effect is not specific to the primary SnapKV-style first-stage
retention rule.
To compare policies with different matched no-repair baselines, the
figure reports score gain over each policy's own matched baseline.
The SnapKV and accumulated-attention rules show large mid-to-high
$K$ gains, and the structurally different sink-plus-recent
rule also passes the gate: at $K=128$, \idlekv{} gains $0.431$ over
matched no-repair while Random-$K$/Oldest-$K$ remain at the matched
baseline.
These are protocol-matched first-stage retention heuristics inspired by
H$_2$O and StreamingLLM, not canonical reproductions of their eviction
kernels, streaming semantics, or serving systems, but they show that the
repair effect is not limited to the primary SnapKV-style retention rule.

\paragraph{Robustness and mechanism.}
The pooled result hides real variation across query partitions, but the
high-$K$ result is not carried by one easy partition.
At $K=128$, all twelve balanced partitions pass the same direction check:
\idlekv{} beats $\bmatch$ and both content-agnostic restore controls.
The minimum score gain over $\bmatch$ is $0.585$ for MQ-NIAH-4Q and
$0.387$ for MQ-NIAH-6Q, and $0.292$ for MQ-NIAH-8Q.
Appendix Figure~\ref{fig:app-partition-endpoints} gives the
per-partition endpoints.
A same-protocol appendix check with Llama-3.1-8B-Instruct on MQ-NIAH-6Q
points in the same direction without supporting a broad model-family
claim: \idlekv{} reaches $1.000$ at $K\in\{64,96,128\}$ versus $0.472$
for matched no-repair, with Random-$K$/Oldest-$K$ at or below matched.

The restored tokens also move the final active cache toward the annotated
$Q_2$ gold spans.
For MQ-NIAH-4Q, \idlekv{} final-active overlap rises from $0.363$ at $K=8$ to
$0.794$ at $K=128$, while $\bmatch$ stays near $0.36$.
For MQ-NIAH-6Q, \idlekv{} overlap rises from $0.498$ to $0.851$, while
$\bmatch$ stays near $0.49$.
The random and oldest-token controls remain near the matched baseline in
both panels.
Overlap is a mechanism diagnostic rather than the task score itself:
the values-only needle-in-a-haystack task can be solved without
recovering every annotated token, which is why \idlekv{} can nearly
saturate quality before final-active overlap reaches the $\goldk$
reference.

\paragraph{Latency accounting.}
We separate quality-linked scorer diagnostics from systems-capacity
measurements.
The exact scorer used in the quality curves is a research diagnostic,
not a deployable implementation: at $K=96$, p50 is $6.08$--$7.62$\,s
across MQ-NIAH-2Q/4Q/6Q/8Q and is dominated by scoring.
The $Q_2$-appended-state proxy reduces paired 4Q/6Q p50 to
$0.83$/$0.71$\,s while still beating $\bmatch$ in paired $n{=}100$
probes.
The systems question is whether a scalable score/select/promote path can
fit inside realistic idle windows, not whether the current exact scorer
is deployable.
Figure~\ref{fig:runtime-envelope} reports that path as a single-node,
component-summed p95 envelope: a chunked GPU selector streams
Qwen-shaped BF16 key rows from pinned host memory, scores a length-64
query, selects $K$ rows, and adds the measured 32K-active
move-and-inject p95 for the same $K$.
The probe is not a joined end-to-end latency distribution and excludes
router decisions, remote or disk fetch, queueing, generation, and
overlap with resumed decoding.
Across $K\in\{96,512,1024,5000\}$, the length-64 p95 curve changes by
at most $6.1$\,ms, so candidate-store size is the dominant scaling axis.
At $K=5000$, p95 grows from $50.0$\,ms at 32K candidates to $1.20$\,s
at 1M and $4.64$\,s at 4M; the 4M endpoint corresponds to about
240\,GB of offloaded Qwen-shaped BF16 KV.
These values sit within the seconds-scale regime reported for
web-agent API/environment waits and coding-agent OS/tool
execution~\citep{agent_efficiency,agentcgroup}, but they are capacity

\section{Discussion and Limitations}
\label{sec:discussion}
% -----------------------------------------------------------------------

\paragraph{Positioning.}
\idlekv{} is not a standalone serving stack.
It is a cache-management operator and benchmark primitive for
resource-adaptive tiered-KV runtimes.
It is complementary to serving systems that preserve, route, offload, or
resume paused requests efficiently~\citep{vllm,infercept,lmcache,ttkv,
continuum,agentserve}: here the question is what quality-improving
promotion policy can run during the interruption.
The setting is also distinct from within-turn query-aware eviction
because the request has already been compressed and full-prefix
recomputation is disallowed.

\paragraph{Applicability and limitations.}
Repair should help when new turn information changes relevance, the base
eviction policy still has real choice beyond recency, and the idle gap is
long enough to cover repair overhead.
The present evidence is intentionally narrow: the main quality figures
use one primary model, one primary first-stage retention rule, one
synthetic split-query family, and fixed full-cache-generated turn-$N$
transcripts.
The central missing systems experiment is trace-scheduled: repair should
be tested against real tool/environment wait distributions, queue state,
and overlap.
We also do not yet include a learned selector, a latency-normalized
full-budget $Q_2$-time reselection frontier, or a full-prefix recompute
baseline.
The curves match the resumed active GPU KV budget, not total
host-plus-device storage.
Figure~\ref{fig:policy-breadth} gives accumulated-attention and
sink-plus-recent first-stage retention checks, and Appendix
Figure~\ref{fig:app-transfer-runtime} gives one same-protocol
Llama-3.1-8B-Instruct MQ-NIAH-6Q portability check, but broader
retention-rule and multi-model transfer remain open.
Because repair assumes an offloaded evicted-KV store, it also creates a
two-level retention problem.
A strict device-resident policy chooses the active KV budget, while a
looser off-device policy decides which evicted rows remain searchable,
compressed, summarized, or recomputable; in our Qwen-shaped accounting,
4M searchable rows already imply about 240\,GB before metadata.
\idlekv{} provides evidence for the middle operation in this hierarchy:
turn-conditioned promotion from the warm tier during an idle gap.
This is a hardware-facing implication rather than a hardware result: the
measurements motivate tiered-KV interfaces that expose scoring,
promotion, and demotion, but they do not establish area, energy, or
bandwidth claims, and \idlekv{} does not solve off-device retention.
Future benchmarks should therefore vary the number, timing, and conflict
structure of new-turn constraints: the 2Q and 8Q breadth checks suggest
that easy regimes can saturate while stricter multi-query turns expose
how quickly eviction failures accumulate.
The locked five-turn diagnostic in Figure~\ref{fig:multiturn} strengthens
the dynamic-workflow evidence, but it remains synthetic and a stale
previous-query scorer still closes part of the gap.
The current selector also leaves visible gap to $\goldk$ at moderate $K$,
which motivates stronger selection rules and broader task families with
the same matched-budget controls.

% -----------------------------------------------------------------------
\section{Conclusion}
\label{sec:conclusion}
% -----------------------------------------------------------------------

Pauses between agent turns can be treated as part of the inference budget
rather than as dead time.
\idlekv{} studies one concrete way to spend that budget: promote
off-device KV back into the active GPU cache before the next turn under
a matched resumed-cache budget.
On calibrated pooled split-query MQ-NIAH-4Q/6Q/8Q panels, the final
exact evaluations show that idle-window repair can substantially
outperform matched no-repair retention while random and oldest-token
restores stay near baseline.
The remaining work is to broaden the evidence beyond this synthetic
family and to close the gap between the current repair selection rule
and the $\goldk$ reference.
More broadly, long-context agent systems may need to be judged not only
by how efficiently they decode, but by how well they maintain KV state
between turns.

\section{Additional Evaluation Details}
\label{app:evaluation-details}

\paragraph{Protocol details.}
Concise split prompts request values only, comma-separated, with a short
answer prefix and fixed decode budgets of 24 tokens for MQ-NIAH-4Q,
48 tokens for MQ-NIAH-6Q, and 64 tokens for MQ-NIAH-8Q.
The exact Q$_2$ attention-query scorer extracts the model's actual
self-attention query vectors for $Q_2$ prompt tokens after RoPE and
scores them against active and evicted keys; this is distinct from
decode-time answer-token attention.
The $Q_2$-appended-state proxy instead appends $Q_2$ to the compressed
cache with an ordinary forward pass and reuses the appended $Q_2$ cache
rows as scoring rows.

\paragraph{Hindsight reference.}
$\goldk$ starts from the same $B$ state and offloaded evicted-KV store
as \idlekv{}.
It enumerates the small family of annotated gold $Q_2$ span-group
subsets, one key/value span group per requested needle, evaluates each
subset once under actual $Q_2$ generation, and chooses the
highest-scoring subset with total cost at most $K$.
It is exact over this benchmark-specific span-group family, but not over
all arbitrary token subsets or possible final generations.

\paragraph{Partition choices.}
For MQ-NIAH-4Q, the pooled panel uses
$14{\rightarrow}23$, $24{\rightarrow}13$, and
$34{\rightarrow}12$.
These are the balanced $2|2$ partitions whose $Q_2$ side excludes the
tail-anchored fourth needle.
For MQ-NIAH-6Q, the pooled panel uses
$156{\rightarrow}234$, $256{\rightarrow}134$,
$356{\rightarrow}124$, and $456{\rightarrow}123$, so the second turn
excludes the two latest needles.
For MQ-NIAH-8Q, the pooled panel uses
$5678{\rightarrow}1234$, $1678{\rightarrow}2345$,
$2678{\rightarrow}1345$, $3678{\rightarrow}1245$, and
$4678{\rightarrow}1235$, again excluding the two latest needles from the
second turn.
Within-turn order is not treated as a separate condition because the
prompt asks for comma-separated values only and scoring checks whether
each requested value is recovered.
Recency-favorable splits such as $12{\rightarrow}34$ are left to
separate diagnostics rather than pooled into the main future-query
repair result.
