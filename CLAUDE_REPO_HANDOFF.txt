High-level project summary
- Project name: IdleKV
- Core idea: use GPU idle time during tool calls in batch-1 LLM agents to repair KV-cache damage caused by eviction.
- Motivation:
  - tool calls often take 2 to 60 seconds
  - during that time the GPU is often idle
  - long-context eviction methods like SnapKV and StreamingLLM save memory by dropping tokens
  - dropped tokens can later become relevant again
  - IdleKV keeps evicted token KV on CPU, re-scores them during idle time, and restores the best ones before the next turn
- Intended claim:
  - not "repair gets more tokens"
  - but "repair picks better tokens at the same final GPU footprint"

Core research framing
- The repo currently follows the matched-footprint framing from instructions.md:
  - Condition A: monolithic pass, no eviction
  - Condition B_onset: compressed-cache baseline at base keep budget B_base
  - Condition B_match: no-repair matched-footprint baseline at B_match = B_base + K_slots
  - Condition C: IdleKV repair, prefill at B_base and restore K_slots during idle time so final footprint is B_match
  - Matched baselines: Random-K and Recency-K at the same final footprint
- Primary later metrics:
  - matched-footprint recovery = (C - B_match) / (A - B_match)
  - selection lift = C - B_match

- Purpose:

  - phases/phase4_eviction_buffer/src/buffer/eviction_buffer.py

  - phases/phase4_eviction_buffer/tests/
- Key implementation details:
  - EvictionBuffer stores one token per entry with per-layer KV on CPU pinned memory
  - selection strategies:
    - l2_norm
    - dot_product
    - random
    - recency_inverse
  - exact CPU scoring is used; no ANN / FAISS / stochastic retrieval is implemented
  - recent_q_vecs are extracted from the active compressed cache via extract_recent_q_vecs
  - end-to-end repair uses inject_kv from Phase 2
  - live profiling uses resume_forward from Phase 2 so cache_position and position_ids are handled correctly
